**Step 2: Write the restore script**

Create /workspace/development/ems-production/scripts/restore-on-first-run.sh with exactly this content:

```bash
#!/bin/bash
# restore-on-first-run.sh
# Called by the configurator service on first boot.
# Checks if the site database already exists in MariaDB.
# If not, restores from the baked-in backup and runs bench migrate.
set -euo pipefail

SITE_NAME="${FRAPPE_SITE_NAME:-university.local}"
DB_NAME="${DB_NAME:-_placeholder}"
BACKUP_FILE="/home/frappe/frappe-bench/db-backup/university_local.sql.gz"

echo "[restore] Checking if site '${SITE_NAME}' database exists in MariaDB at ${DB_HOST}:${DB_PORT}..."

DB_EXISTS=$(mysql \
  -h "${DB_HOST}" \
  -P "${DB_PORT:-3306}" \
  -u root \
  -p"${MYSQL_ROOT_PASSWORD}" \
  --skip-column-names \
  -e "SELECT COUNT(*) FROM information_schema.SCHEMATA WHERE SCHEMA_NAME='${DB_NAME}';" \
  2>/dev/null || echo "0")

if [ "${DB_EXISTS}" = "0" ] || [ -z "${DB_EXISTS}" ]; then
  if [ -f "${BACKUP_FILE}" ]; then
    echo "[restore] First run detected. Restoring database from backup: ${BACKUP_FILE}"
    cd /home/frappe/frappe-bench
    bench restore "${BACKUP_FILE}" \
      --mariadb-root-password "${MYSQL_ROOT_PASSWORD}" \
      --site "${SITE_NAME}" \
      --force
    echo "[restore] Running bench migrate to apply schema changes..."
    bench --site "${SITE_NAME}" migrate
    echo "[restore] Database restore and migration complete."
  else
    echo "[restore] WARNING: No backup file found at ${BACKUP_FILE}. Skipping restore."
    echo "[restore] You must manually create the site: bench new-site ${SITE_NAME}"
  fi
else
  echo "[restore] Database already exists. Skipping restore."
fi
```

Make it executable: `chmod +x /workspace/development/ems-production/scripts/restore-on-first-run.sh`

Find the RUN block ending with `find apps -mindepth 1 -path "*/.git" | xargs rm -fr`. Insert the following IMMEDIATELY AFTER that block and BEFORE the line `FROM base AS backend`:

CRITICAL: These three blocks must be in the builder stage (before `FROM base AS backend`), not in the backend stage. The `COPY --from=builder` in the backend stage will then carry them into the final image.

**Edit B — Remove sites/assets from the VOLUME declaration in the backend stage.**

In the backend stage (after `FROM base AS backend`), find the VOLUME instruction. It will look something like:

```dockerfile
VOLUME [ \
  "/home/frappe/frappe-bench/sites", \
  "/home/frappe/frappe-bench/sites/assets", \
  ...
]
```

Remove `"/home/frappe/frappe-bench/sites/assets"` from this VOLUME list. Keep all other entries unchanged.

WHY: A VOLUME declaration causes Docker to create a new empty mount point at runtime, shadowing any content written to that path during the build stage. The `bench build --production` in Edit A writes compiled JS/CSS into sites/assets/. If sites/assets remains in VOLUME, those assets are invisible at runtime — the Frappe desk UI fails to load. Only the bare `sites/` volume (for runtime persistence of site config and uploads) should remain.
