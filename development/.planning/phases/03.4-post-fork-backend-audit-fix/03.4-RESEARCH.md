### Workflow Architecture
```
Frappe Workflow Document Structure:
{
    "doctype": "Workflow",
    "workflow_name": "Descriptive Name",
    "document_type": "Target DocType",
    "is_active": 1,
    "workflow_state_field": "workflow_state",
    "states": [
        {"state": "State Name", "doc_status": "0|1|2", "allow_edit": "Role"}
    ],
    "transitions": [
        {"state": "From", "action": "Action", "next_state": "To", "allowed": "Role"}
    ]
}
```

### Pattern: Workflow State Field
Frappe Workflow requires a `workflow_state` field on the target DocType. If the DocType doesn't have one, Frappe adds it automatically when the Workflow is activated. However, for doctypes that are NOT submittable (like Grievance), workflows use doc_status=0 for all states and track state purely through workflow_state.

### Anti-Patterns to Avoid
- **Creating workflows for child table doctypes**: Child tables (istable=1) cannot have workflows. Only standalone doctypes.
- **Skipping Workflow State creation**: Each workflow state name must exist as a Workflow State document. Create them before the Workflow.
- **Forgetting Workflow Action Master**: Each action name must exist as a Workflow Action Master document.
- **Testing workflows with Administrator**: Administrator bypasses workflow checks. Always test with role-specific users.

## Common Pitfalls

### Pitfall 1: University Accounts Settings Not Configured
**What goes wrong:** Fee submission triggers GL posting hook, but University Accounts Settings has no account values configured, causing "Account is mandatory" errors.
**Why it happens:** The singleton doctype exists but was never populated with actual GL Account links after the fork.
**How to avoid:** Seed script MUST configure University Accounts Settings with valid accounts before creating any Fee records. Create fee_collection_account, fee_income_account, fee_receivable_account etc. in the Chart of Accounts first.
**Warning signs:** "Accounts Settings not found" or "Account is mandatory" errors during fee submission.

### Pitfall 2: Workflow States Not Pre-Created
**What goes wrong:** Creating a Workflow document fails because referenced Workflow State or Workflow Action Master documents don't exist.
**Why it happens:** Frappe Workflow expects Workflow State and Workflow Action Master as Link fields. If "Pending Warden Approval" isn't a Workflow State document, the workflow won't save.
**How to avoid:** Create all Workflow State and Workflow Action Master documents BEFORE creating Workflow documents. The fee_refund_workflow.json fixture shows the correct pattern -- it includes both.
**Warning signs:** LinkValidationError when saving Workflow.

### Pitfall 3: Workflows on Non-Submittable Doctypes
**What goes wrong:** Workflow with doc_status=1 (Submitted) on a non-submittable doctype causes errors.
**Why it happens:** Grievance (is_submittable=0) can't use doc_status=1 transitions. The workflow must use doc_status=0 for all states.
**How to avoid:** Check is_submittable before designing workflow states. For non-submittable doctypes, ALL states must have doc_status=0.
**Warning signs:** "This document cannot be submitted" errors during workflow transition.

### Pitfall 4: Report Module Mismatch Post-Fork
**What goes wrong:** Reports registered in DB under "Accounts" module but the actual Python code is now in "University Finance" module.
**Why it happens:** Phase 03.3 fork copied report code but report registration in the database may still point to old module.
**How to avoid:** The report audit script tests execution, which will catch import failures. Any university_finance report that fails with ModuleNotFoundError needs its DB registration updated.
**Warning signs:** "Module not found" or "Report not found" errors.

### Pitfall 5: Demo Data Foreign Key Dependencies
**What goes wrong:** Creating a Fee record fails because the student's Program Enrollment doesn't exist yet, or the fee component's GL Account doesn't exist.
**Why it happens:** Frappe validates Link fields. Demo data must be created in dependency order.
**How to avoid:** Follow strict creation order: Company -> Academic Year -> Departments -> Programs -> Courses -> Employees -> Students -> Enrollments -> Fee Structure -> Fees. Use `ignore_links=True` sparingly.
**Warning signs:** LinkValidationError, MandatoryError during seed.

### Pitfall 6: Duplicate Workflow for Same DocType
**What goes wrong:** Two active workflows on the same doctype causes unpredictable behavior.
**Why it happens:** Creating a new workflow without checking if one already exists.
**How to avoid:** Always query `frappe.get_all("Workflow", {"document_type": dt, "is_active": 1})` before creating.
**Warning signs:** "Multiple active workflows found" warning or unexpected state transitions.

## Code Examples

### Creating a Workflow Programmatically
```python
# Source: Existing fee_refund_workflow.json fixture pattern
import frappe

def create_workflow(name, document_type, states, transitions):
    """Create a workflow if it doesn't already exist."""
    existing = frappe.db.exists("Workflow", {"workflow_name": name})
    if existing:
        return existing

    # Ensure all Workflow States exist
    for s in states:
        if not frappe.db.exists("Workflow State", s["state"]):
            frappe.get_doc({
                "doctype": "Workflow State",
                "workflow_state_name": s["state"],
                "style": s.get("style", "")
            }).insert(ignore_permissions=True)

    # Ensure all Workflow Action Masters exist
    for t in transitions:
        if not frappe.db.exists("Workflow Action Master", t["action"]):
            frappe.get_doc({
                "doctype": "Workflow Action Master",
                "workflow_action_name": t["action"]
            }).insert(ignore_permissions=True)

    wf = frappe.get_doc({
        "doctype": "Workflow",
        "workflow_name": name,
        "document_type": document_type,
        "is_active": 1,
        "send_email_alert": 0,
        "workflow_state_field": "workflow_state",
        "states": [{"doctype": "Workflow Document State", **s} for s in states],
        "transitions": [{"doctype": "Workflow Transition", **t} for t in transitions],
    })
    wf.insert(ignore_permissions=True)
    return wf.name
```

```python
# Source: university_accounts_settings.json doctype structure
import frappe

def configure_accounts_settings(company):
    """Configure University Accounts Settings for GL posting."""
    settings = frappe.get_single("University Accounts Settings")
    settings.company = company
    settings.auto_post_gl = 1

    # These accounts must exist in the Chart of Accounts
    settings.fee_collection_account = f"Fee Collection - {company[:3]}"
    settings.fee_income_account = f"Fee Income - {company[:3]}"
    settings.fee_receivable_account = f"Student Receivable - {company[:3]}"
    settings.save(ignore_permissions=True)
```

### Restricting ERPNext Accounts Reports via Permissions
```python
# Source: Frappe permission API
import frappe

def restrict_erpnext_accounts_reports():
    """Restrict original ERPNext Accounts reports to System Manager only."""
    # Reports in 'Accounts' module that have university_finance equivalents
    duplicate_reports = [
        "Accounts Receivable", "Accounts Payable", "General Ledger",
        "Trial Balance", "Balance Sheet", "Profit and Loss Statement",
        "Accounts Receivable Summary", "Accounts Payable Summary",
    ]
    for report_name in duplicate_reports:
        if frappe.db.exists("Report", report_name):
            # Set report as restricted
            frappe.db.set_value("Report", report_name, "is_standard", "No")
            # Remove all existing permissions except System Manager
            frappe.db.sql("""
                DELETE FROM `tabHas Role`
                WHERE parent=%s AND parenttype='Report'
                AND role != 'System Manager'
            """, report_name)
```

   - What's unclear: Whether the original ERPNext reports still reference erpnext.accounts or if they now break post-fork.

3. **Workflow for Grievance (Non-Submittable)**
   - What we know: Grievance is NOT submittable (is_submittable=0) but has a rich status field (Draft through Closed with 10 states).
   - What's unclear: Whether Frappe Workflow on non-submittable doctype will conflict with existing status field logic in grievance.py.
   - Recommendation: Use doc_status=0 for all workflow states. Map workflow_state to status field via update_field/update_value in workflow states. Test carefully since grievance.py may have custom status logic.

4. **bench migrate Safety Post-Workflow-Creation**
   - What we know: Phase 03.3 confirmed bench migrate works cleanly with 179 doctypes.
   - What's unclear: Whether adding 10+ new Workflow documents will require another bench migrate, or if they're pure data documents.
