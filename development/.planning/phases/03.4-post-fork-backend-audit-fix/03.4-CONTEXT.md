# Phase 03.4: Post-Fork Backend Audit & Fix - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning

<domain>
## Phase Boundary

After the accounts fork (Phase 03.3), the entire ERP backend is fully re-tested — every doctype accessible in Frappe Desk, every report running, every workflow functional, all cross-app connections verified — ensuring complete backend stability before any frontend portal work begins. Also seeds comprehensive demo data and creates all missing workflows across the app.

</domain>

<decisions>
## Implementation Decisions

### Demo Data
- **D-01:** Large-scale data: 500+ students across multiple programs, 50+ faculty, multiple departments, full academic year with terms
- **D-02:** Wipe existing seed data before re-seeding (clean slate approach)
- **D-03:** Financial transactions MUST go through the new university_finance module — fee records that post GL entries via the forked module to validate the fork works end-to-end with real data
- **D-04:** Script must be idempotent (check-before-create pattern, safe to re-run after fixes)
- **D-05:** Script approach: Claude decides whether to extend existing `seed_demo_data.py` or create new dedicated script based on what makes sense

### Workflow Testing & Creation
- **D-06:** Full end-to-end testing in Frappe Desk — actually submit documents through each workflow stage (Draft→Submit→Approve) via bench console, verify state transitions, GL postings, status updates
- **D-07:** Discover ALL needed workflows across the entire app — not just the 4 listed (fee collection, leave approval, admission, grade submission) but also hostel fees, room allocation, grievance workflow, transport, library, placement, and any others identified during research
- **D-08:** Create any missing workflows that should exist but don't (hostel fee payment, room allocation approval, grievance resolution, etc.)
- **D-09:** Test ALL workflows — both existing and newly created
- **D-10:** Fix failures immediately — no documenting and deferring. If a workflow fails, find and fix the root cause in this phase

### Report Failure Handling
- **D-11:** Fix university_erp and university_finance reports only. ERPNext/HRMS reports that fail due to missing Company setup are expected failures in a university context — leave them as-is
- **D-12:** Disable original ERPNext Accounts report versions via permissions — restrict so only System Manager can see them. University users see university_finance versions (Fee Receivable, Vendor Payable, Student Ledger, etc.)

### Claude's Discretion
- Exact demo data distribution (how many students per program, department sizes)
- Which specific workflows to create beyond the obvious ones (based on research of the app modules)
- How to structure the seed script (extend vs. new file)
- Order of audit operations (reports first vs. doctypes first vs. workflows first)

</decisions>

<specifics>
## Specific Ideas

- User wants workflows across ALL modules — hostel, transport, library, placement, grievance — not just the core academic ones
- Demo data should make every Desk list view show meaningful content (not just 1-2 records)
- Financial demo data validates the Phase 03.3 fork actually works in practice
- This phase is a quality gate before any frontend portal work begins

</specifics>
