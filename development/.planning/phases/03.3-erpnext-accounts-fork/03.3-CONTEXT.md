# Phase 03.3: ERPNext Accounts Module Fork - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Fork the entire ERPNext Accounts module (179 doctypes, 51 reports, all controllers) into university_erp as a new `university_finance` module. Relabel all user-facing elements with university terminology. Archive unused doctypes. Set up university-specific financial workflows. Goal: ERPNext will eventually be removed — the fork must be fully standalone. All existing finance flows (fee→GL, payment gateway, bank reconciliation) must work through the forked module.

</domain>

<decisions>
## Implementation Decisions

### Fork Scope (D-01)
- **Full module fork** — copy all 179 doctypes, 51 reports, all controllers (general_ledger.py, utils.py, party.py, deferred_revenue.py) into university_erp
- **New module inside university_erp** — created as `university_erp/university_finance/` (not a separate app)
- **ERPNext removal goal** — the fork must be fully standalone. All references to `erpnext.accounts` must be redirected to `university_erp.university_finance`
- **Currently only 1 direct import** from erpnext.accounts (`make_gl_entries` in gl_posting.py) — but since ERPNext will be removed, ALL accounting functionality must be in the fork

### Relabeling Strategy (D-02)
- **Full university rebrand** — rename both user-facing labels AND module name:
  - Customer → Student
  - Sales Invoice → Fee Invoice
  - Supplier → Vendor
  - Sales Order → Fee Order
  - Purchase Invoice → Procurement Invoice
  - Purchase Order → Purchase Requisition
  - Module name: Accounts → University Finance
- **Internal accounting terms unchanged** — GL Entry, Journal Entry, Cost Center, Account, Fiscal Year keep their names (they're standard accounting concepts)
- **Implementation approach** — rename doctype labels, field labels, workspace labels. Keep underlying table names (e.g., `tabGL Entry`) to avoid migration complexity

### Unused Doctype Handling (D-03)
- **Archive approach** — move unused doctypes to `university_finance/_archived/` directory
- **Unused doctypes include**: POS Invoice, POS Profile, POS Settings, POS Closing Entry, Dunning, Subscription, Subscription Plan, Share Transfer, Shareholder, Loyalty Program, Loyalty Point Entry, Coupon Code, Shipping Rule, Promotional Scheme, Sales Partner Item, Territory Item
- **Keep for potential use**: Budget, Payment Request, Bank Guarantee, Exchange Rate Revaluation, Payment Terms Template, Cheque Print Template

### Workflow Design (D-04)
- **Workflow decisions deferred to planning** — Claude determines which workflows are needed based on forked doctypes
- **Existing ERPNext workflows**: review case-by-case during planning (keep accounting workflows like Journal Entry/Payment Entry approval; remove sales/purchase/POS workflows)
- **Financial approval role**: University Finance Officer is the sole approver for all financial transactions
- **Existing university_erp workflows preserved**: leave_application_workflow, teaching_assignment_workflow, fee_refund_workflow

### Claude's Discretion
- Exact list of which doctypes to archive vs keep (beyond the ones explicitly listed)
- Workflow state names and transitions
- Which ERPNext reports are relevant for university context
- How to handle ERPNext's accounting dimensions in university context
- Whether to rename table names or just labels (recommendation: labels only to avoid migration)
- Test strategy for verifying GL posting works after fork

</decisions>
