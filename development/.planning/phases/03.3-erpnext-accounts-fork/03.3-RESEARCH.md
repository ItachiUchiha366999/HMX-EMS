# Phase 03.3: ERPNext Accounts Module Fork - Research

**Researched:** 2026-03-22
**Domain:** Frappe module architecture, ERPNext Accounts internals, Python module forking
**Confidence:** HIGH (based on direct source code analysis of the actual codebase)

## Summary

This phase requires forking the entire ERPNext Accounts module (179 doctypes across 87 standalone + 92 child tables, 56 reports, ~5,000 lines of core Python utilities, and ~34,000 lines of doctype controllers) into the `university_erp` app as a new `university_finance` module. The fork is the first step toward eventual ERPNext removal.

The key insight from research is that the fork MUST NOT copy Sales Invoice, Purchase Invoice, or POS-related doctypes with their original controllers intact, because those controllers inherit from `SellingController`/`BuyingController` which pull in the entire stock/selling/buying module chains. Instead, the fork strategy should: (1) copy all 179 doctypes as file directories, (2) for the ~56 archivable doctypes (POS, Dunning, Subscription, Share, Loyalty, etc.), move to `_archived/`, (3) for Sales Invoice and Purchase Invoice, strip the selling/buying controller inheritance and replace with a simplified university-specific controller inheriting from `AccountsController`, (4) rewrite all `from erpnext.accounts.*` imports to `from university_erp.university_finance.*`, and (5) copy the ERPNext root-level utility functions (`get_default_company`, `get_company_currency`, etc.) into a `university_finance/_erpnext_compat.py` shim.

**Primary recommendation:** Fork in phases -- first copy files and fix module registration, then rewrite imports, then relabel, then archive. Do NOT try to do everything in one pass. The `bench migrate` step is the riskiest moment and must happen after all JSON `"module"` fields are updated.

### Pattern 1: Import Path Rewriting
**What:** Every `from erpnext.accounts.*` import inside the forked code must be rewritten to `from university_erp.university_finance.*`
**When to use:** All forked .py files
**Example:**
```python
# BEFORE (in erpnext/accounts/general_ledger.py):
import erpnext
from erpnext.accounts.doctype.accounting_dimension.accounting_dimension import get_accounting_dimensions
from erpnext.accounts.utils import create_payment_ledger_entry, is_immutable_ledger_enabled

# AFTER (in university_erp/university_finance/general_ledger.py):

    get_default_company, get_company_currency, get_default_cost_center

from university_erp.university_finance.doctype.accounting_dimension.accounting_dimension import get_accounting_dimensions
```

### Pattern 2: ERPNext Compatibility Shim
**What:** Copy the ~12 utility functions from `erpnext/__init__.py` into a compatibility module
**When to use:** All forked code that does `import erpnext` or `from erpnext import *`
**Example:**
```python
# university_erp/university_finance/_erpnext_compat.py
import frappe
from frappe.utils.user import is_website_user

def get_default_company(user=None):
    """Get default company for user"""
    from frappe.defaults import get_user_default_as_list
    if not user:
        user = frappe.session.user
    companies = get_user_default_as_list("company", user)
    if companies:
        return companies[0]
    return frappe.db.get_single_value("Global Defaults", "default_company")

def get_company_currency(company):
    """Returns the default company currency"""
    if not frappe.flags.company_currency:
        frappe.flags.company_currency = {}
    if company not in frappe.flags.company_currency:
        frappe.flags.company_currency[company] = frappe.db.get_value(
            "Company", company, "default_currency", cache=True
        )
    return frappe.flags.company_currency[company]

def get_default_cost_center(company):
    if not company:
        return None
    if not frappe.flags.company_cost_center:
        frappe.flags.company_cost_center = {}
    if company not in frappe.flags.company_cost_center:
        frappe.flags.company_cost_center[company] = frappe.get_cached_value(
            "Company", company, "cost_center"
        )
    return frappe.flags.company_cost_center[company]

def get_default_finance_book(company=None):
    if not company:
        company = get_default_company()
    if not hasattr(frappe.local, "default_finance_book"):
        frappe.local.default_finance_book = {}
    if company not in frappe.local.default_finance_book:
        frappe.local.default_finance_book[company] = frappe.get_cached_value(
            "Company", company, "default_finance_book"
        )
    return frappe.local.default_finance_book[company]

def get_party_account_type(party_type):
    if not hasattr(frappe.local, "party_account_types"):
        frappe.local.party_account_types = {}
    if party_type not in frappe.local.party_account_types:
        frappe.local.party_account_types[party_type] = (
            frappe.db.get_value("Party Type", party_type, "account_type") or ""
        )
    return frappe.local.party_account_types[party_type]

# allow_regional decorator - simplified for university (no regional overrides needed)
import functools
def allow_regional(fn):
    @functools.wraps(fn)
    def caller(*args, **kwargs):
        return fn(*args, **kwargs)
    return caller
```

### Pattern 3: Doctype JSON Module Field Update
**What:** Every doctype JSON file has `"module": "Accounts"` which must change to `"module": "University Finance"`
**When to use:** All 179 doctype JSON files
**Critical detail:** This field determines which module "owns" the doctype in Frappe's DB. During `bench migrate`, Frappe reads this JSON and updates the DocType record's module field.
```python
# Script to update all doctype JSONs:
import json, os, glob

base = "university_erp/university_finance/doctype"
for json_file in glob.glob(f"{base}/*/*.json"):
    with open(json_file) as f:
        doc = json.load(f)
    if doc.get("module") == "Accounts":
        doc["module"] = "University Finance"
        with open(json_file, "w") as f:
            json.dump(doc, f, indent=1, sort_keys=True)
```

### Anti-Patterns to Avoid
- **Copying test files:** Do NOT copy the ~200+ test files from `erpnext/accounts/doctype/*/test_*.py`. They depend on the entire ERPNext test infrastructure, fixture setup, and factory functions. Write fresh focused tests instead.
- **Renaming DB table names:** Do NOT change table names (e.g., `tabGL Entry` stays `tabGL Entry`). This would require data migration, index recreation, and breaks all existing data. Only change labels.
- **Modifying ERPNext source:** Do NOT modify files in the `erpnext` app directory. The fork must be entirely within `university_erp`.
- **Single massive commit:** Do NOT try to fork everything in one commit. Break into: (1) file copy, (2) JSON module update, (3) import rewriting, (4) controller adaptation, (5) relabeling, (6) archiving.
- **Keeping SellingController/BuyingController inheritance:** Sales Invoice inherits from `SellingController(StockController)` which imports 15+ stock/selling modules. Fee Invoice MUST NOT keep this inheritance -- replace with `AccountsController`.

## Common Pitfalls

### Pitfall 1: Circular Import Between Forked Modules
**What goes wrong:** `general_ledger.py` imports from `utils.py` which imports from `doctype/account/account.py` which imports from `utils.py`. After forking, Python circular imports can deadlock.
**Why it happens:** ERPNext uses lazy imports (inside functions) to break cycles, but when paths change, the lazy import targets change too.
**How to avoid:** Rewrite imports methodically file-by-file. Test each core file can be imported standalone: `python -c "from university_erp.university_finance.general_ledger import make_gl_entries"`
**Warning signs:** `ImportError: cannot import name X from partially initialized module Y`

### Pitfall 2: Module Def Not Created in Database
**What goes wrong:** Frappe resolves modules via `Module Def` records in the database. Having `University Finance` in `modules.txt` creates the Module Def on install, but bench migrate also needs the doctype JSONs to have the correct module name.
**Why it happens:** Frappe's `get_doctype_module()` queries the DocType table's `module` field. If doctype JSONs still say `"module": "Accounts"`, the fork won't work.
**How to avoid:** (1) Verify `Module Def` for `University Finance` exists in DB, (2) Update ALL doctype JSONs before running `bench migrate`, (3) After migrate, verify: `frappe.db.get_value("DocType", "GL Entry", "module")` returns "University Finance"
**Warning signs:** `DocType not found` errors, `Module not found` errors

### Pitfall 3: Duplicate DocType Names Between ERPNext and Fork
**What goes wrong:** If you fork GL Entry into university_finance but ERPNext's GL Entry still exists with module=Accounts, Frappe sees two DocType definitions for the same name. During migrate, the LAST app to sync wins.
**Why it happens:** Frappe syncs doctypes per-app in app install order. ERPNext and university_erp both define "GL Entry".
**How to avoid:** Do NOT create duplicate doctype names. The forked doctypes MUST keep the same names (GL Entry, Payment Entry, etc.) because the DB tables already exist. The fork changes the `module` owner from `Accounts` to `University Finance`, but there is only ONE "GL Entry" doctype in the system. After fork + migrate, the DocType row for "GL Entry" will show `module = University Finance` because university_erp is installed after erpnext and its JSON wins.
**Critical insight:** This is actually the CORRECT behavior -- university_erp loads after erpnext, so its doctype definitions override. When ERPNext is eventually removed, the university_finance definitions are the only ones left.
**Warning signs:** During `bench migrate`, warnings about "DocType GL Entry already synced from Accounts" -- these are expected and OK.

### Pitfall 4: AccountsController Dependencies on Stock/Selling/Buying
**What goes wrong:** `AccountsController` (4188 lines) imports from `erpnext.buying.utils`, `erpnext.stock.get_item_details`, `erpnext.stock.doctype.packed_item`, etc. Forking it means pulling in half of ERPNext.
**Why it happens:** AccountsController is shared by ALL transaction documents (invoices, payments, journal entries). It handles item pricing, tax calculation, stock reservation -- features a university doesn't need.
**How to avoid:** Fork AccountsController but aggressively strip stock/selling/buying methods. University needs: GL entry methods, tax/charge methods, payment term methods, accounting dimension methods. University does NOT need: item pricing, packed items, stock reservation, purchase rate updates, serial numbers.
**Warning signs:** ImportErrors when running `bench migrate` -- trace the import chain to find stock/selling/buying dependencies.

### Pitfall 5: `erpnext` Root Module Import
**What goes wrong:** Many files do `import erpnext` and then call `erpnext.get_company_currency()`, `erpnext.get_default_company()`, etc. After removing ERPNext, these break.
**Why it happens:** ERPNext's `__init__.py` has 12 utility functions that are used everywhere in the accounts module.
**How to avoid:** Create `_erpnext_compat.py` with these functions. In forked files, replace `import erpnext` with `from university_erp.university_finance import _erpnext_compat as erpnext` -- this is a one-line change per file that preserves all downstream `erpnext.function()` calls.
**Warning signs:** `ModuleNotFoundError: No module named 'erpnext'` after ERPNext removal

### Pitfall 6: Sales Invoice / Purchase Invoice Controller Inheritance
**What goes wrong:** Sales Invoice inherits from `SellingController(StockController)`, not `AccountsController`. If you fork it as-is, it drags in the entire selling/stock module chain: Item, Warehouse, Delivery Note, Customer Group, Territory, Price List, Serial No, etc.
**Why it happens:** In ERPNext, Sales Invoice is a selling document that happens to create GL entries. In university context, Fee Invoice is a finance document period.
**How to avoid:** When forking Sales Invoice -> Fee Invoice, change the inheritance from `SellingController` to `AccountsController`. Strip all selling/stock methods. Keep only: GL entry creation, payment allocation, tax handling, accounting dimension support. Same for Purchase Invoice -> Procurement Invoice: change from `BuyingController` to `AccountsController`.
**Warning signs:** 15+ ImportErrors tracing back to selling_controller -> stock_controller -> stock modules

### Pitfall 7: Report Python Files Import erpnext Extensively
**What goes wrong:** Every report's `execute()` function imports from various erpnext modules. The `financial_statements.py` shared utility imports from `erpnext.accounts.utils`, `erpnext.accounts.doctype.accounting_dimension`, etc.
**Why it happens:** Reports are standalone scripts that fetch data -- they import whatever they need.
**How to avoid:** Fork `report/financial_statements.py` and `report/utils.py` as shared utilities. For each kept report, rewrite imports. This is mechanical but tedious -- ~26 reports * ~5 imports each = ~130 import rewrites.
**Warning signs:** Report execution errors visible in Frappe desk when opening any financial report.

### Copy and Module-Rewrite Script
```python
#!/usr/bin/env python3
"""
Fork ERPNext Accounts module into university_finance.
Run from frappe-bench/apps/university_erp/ directory.
"""
import json
import os
import re
import shutil

SRC = "../erpnext/erpnext/accounts"
DST = "university_erp/university_finance"

def copy_doctype_dirs():
    """Copy all doctype directories"""
    src_dt = os.path.join(SRC, "doctype")
    dst_dt = os.path.join(DST, "doctype")

    for d in os.listdir(src_dt):
        src_path = os.path.join(src_dt, d)
        dst_path = os.path.join(dst_dt, d)
        if os.path.isdir(src_path) and not d.startswith("__"):
            if os.path.exists(dst_path):
                shutil.rmtree(dst_path)
            shutil.copytree(src_path, dst_path)
            # Remove test files and __pycache__
            for root, dirs, files in os.walk(dst_path):
                for f in files:
                    if f.startswith("test_") or f == "__pycache__":
                        os.remove(os.path.join(root, f))
                for d_name in dirs:
                    if d_name == "__pycache__":
                        shutil.rmtree(os.path.join(root, d_name))

def update_module_in_jsons():
    """Update 'module' field in all doctype JSONs"""
    dst_dt = os.path.join(DST, "doctype")
    for d in os.listdir(dst_dt):
        json_file = os.path.join(dst_dt, d, f"{d}.json")
        if os.path.exists(json_file):
            with open(json_file) as f:
                doc = json.load(f)
            if doc.get("module") == "Accounts":
                doc["module"] = "University Finance"
                with open(json_file, "w") as f:
                    json.dump(doc, f, indent=1, sort_keys=True)

def rewrite_imports_in_file(filepath):
    """Rewrite erpnext.accounts imports to university_finance"""
    with open(filepath) as f:
        content = f.read()

    # Replace erpnext.accounts -> university_erp.university_finance
    content = content.replace(
        "from erpnext.accounts.",
        "from university_erp.university_finance."
    )
    content = content.replace(
        "import erpnext.accounts.",
        "import university_erp.university_finance."
    )

    # Replace bare 'import erpnext' with compat shim
    content = re.sub(
        r'^import erpnext$',
        'from university_erp.university_finance import _erpnext_compat as erpnext',
        content,
        flags=re.MULTILINE
    )
    content = re.sub(
        r'^from erpnext import (.+)$',
        r'from university_erp.university_finance._erpnext_compat import \1',
        content,
        flags=re.MULTILINE
    )

    # Replace erpnext.exceptions -> university_finance._exceptions
    content = content.replace(
        "from erpnext.exceptions",
        "from university_erp.university_finance._exceptions"
    )

    # Replace erpnext.controllers -> university_finance.controllers
    content = content.replace(
        "from erpnext.controllers.",
        "from university_erp.university_finance.controllers."
    )
    content = content.replace(
        "from erpnext.utilities.transaction_base",
        "from university_erp.university_finance.controllers.transaction_base"
    )
    content = content.replace(
        "from erpnext.utilities.regional",
        "from university_erp.university_finance._erpnext_compat"
    )

    with open(filepath, "w") as f:
        f.write(content)
```

## Open Questions

1. **Will bench migrate handle 123+ new doctype definitions gracefully?**
   - What we know: Frappe syncs all doctype JSONs during migrate. With 123 doctypes already existing in DB (from ERPNext), the migrate will UPDATE their module field.
   - What's unclear: Whether the migrate will emit warnings or errors about "duplicate" definitions.
   - Recommendation: Test with a backup. Run `bench --site university.local migrate` and capture all output. If it fails, restore and adjust.

2. **How will existing GL Entry data behave after module change?**
   - What we know: GL Entry rows in DB reference account names and voucher types, not module paths. The module field on DocType is metadata, not data.
   - What's unclear: Whether any Frappe framework code uses `doctype.module` at runtime for data operations (vs just admin/discovery).
   - Recommendation: LOW risk. The module field affects admin UI routing, not data operations. Existing GL entries will continue to work.

3. **AccountsController stripping -- how much can be removed?**
   - What we know: 4188 lines with ~30 import lines from stock/selling/buying. Many methods are general-purpose (apply taxes, validate accounts) and needed.
   - What's unclear: Exact method-level dependency map of which methods call stock/selling functions.
   - Recommendation: Fork first with all code, then progressively stub/remove stock imports. Use lazy imports (inside functions) where possible so unused code paths don't fail at import time.

4. **Sales Invoice -> Fee Invoice: label or rename?**
   - What we know: CONTEXT.md says "keep underlying table names". Sales Invoice is `tabSales Invoice` in DB.
   - What's unclear: How to present "Fee Invoice" in desk while keeping `tabSales Invoice` table.
   - Recommendation: Keep doctype name as "Sales Invoice" (table stays `tabSales Invoice`), but update workspace label to "Fee Invoice" and add `label_override` in the workspace JSON. This preserves all existing data and references while showing the university-friendly name in the UI.
