## Decisions Made

1. **SQL .format() pattern for conditions is acceptable** -- The Frappe report pattern of building a conditions string with `%(name)s` placeholders and passing the actual values dict as the second argument to `frappe.db.sql()` is safe. Converting f-string to `.format()` preserves this pattern while removing the f-string scanner flag.

2. **INFO-level SQL findings left unfixed** -- Table name interpolation in audit scripts (audit_cross_app_links.py) and seed scripts (seed_demo_data.py) uses values from controlled sources (disk scan, hardcoded list), not user input. Documented with justification in SQL_AUDIT.md.

3. **Education is authoritative for core data models** -- Fees, Assessment Result, Student Attendance, and Student doctypes all have Education as the authoritative source. university_erp extends via `override_doctype_class` (EXTEND pattern) or creates records via API (CREATE-INTO pattern).

4. **Attendance domain has MEDIUM risk** -- Dual input channels (Student Attendance Tool vs faculty_api.submit_attendance) could create duplicates if idempotency check is bypassed. Currently mitigated by `frappe.db.count` check in faculty_api.
