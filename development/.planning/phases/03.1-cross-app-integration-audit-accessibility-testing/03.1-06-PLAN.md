    Verifies all Frappe Desk workspace links resolve to real doctypes/pages.

    4. For each shortcut/link:
       - If type == "DocType": verify `frappe.db.exists("DocType", link_to)` returns True
       - If type == "Report": verify `frappe.db.exists("Report", link_to)` returns True
       - If type == "Page": verify `frappe.db.exists("Page", link_to)` returns True
       - If type == "Dashboard": verify `frappe.db.exists("Dashboard", link_to)` returns True
    5. Also check university_erp Dashboard pages: `frappe.get_all("Dashboard", filters={"module": ["in", university_erp_modules]})` and verify each dashboard's chart/number card references exist.

    **Script 2: audit_sql_injection.py**
    Scans all university_erp .py files for SQL injection patterns.

    1. Walk `university_erp/university_erp/` directory tree.
    2. For each `.py` file, read content and search for dangerous patterns:
       - **Pattern A (f-string)**: regex `r'frappe\.db\.sql\s*\(\s*f["\']'` -- f-string directly in frappe.db.sql()
       - **Pattern B (f-string variable)**: Look for variables assigned f-strings that are later passed to `frappe.db.sql()`. Use a two-pass approach: (1) find all `frappe.db.sql(variable_name` calls, (2) check if `variable_name` was assigned via f-string.
       - **Pattern C (% format)**: regex `r'frappe\.db\.sql\s*\(\s*["\'].*%s.*["\']\s*%\s'` -- old-style % formatting
       - **Pattern D (.format())**: regex `r'frappe\.db\.sql\s*\(.*\.format\('` -- .format() calls
    3. For each finding, record: file_path, line_number, the offending line, pattern type.
    4. Classify severity:
       - CRITICAL: f-string/format with user-provided filter values (from function parameters named `filters`)
       - WARNING: f-string/format with internal values (from frappe.db lookups)
       - INFO: f-string/format used only for table names or column names (less exploitable but still bad practice)
    5. Generate SQL_AUDIT.md:
       - Summary: files scanned, total findings, by severity
       - Table: | # | File | Line | Severity | Pattern | Code Snippet |
       - Recommendations section

    **Script 3: audit_business_logic_duplicates.py**
    Audits overlapping business logic between Education/ERPNext and university_erp.

    1. Analyze 4 known overlap areas:

       **Fees overlap:**
       - Education: `Fees` doctype (fee creation, collection, GL posting via `erpnext.accounts`)
       - university_erp: `Bulk Fee Generator`, `Fee Refund`, `Fee Payment Order`, `fee_collection_summary` report, `fee_defaulters` report
       - university_erp overrides: Check `university_erp/overrides/fees.py` for what it overrides
       - Question: Does university_erp extend Education's Fees, or replace it?
       - Check hooks.py for `doc_events` on `Fees` doctype

       **Exams overlap:**
       - Education: `Assessment Plan`, `Assessment Result`, `Assessment Criteria`, `Assessment Group`
       - university_erp: `Online Examination`, `Internal Assessment`, `Student Exam Attempt`, `Question Bank`, `Answer Sheet`, `Revaluation Request`, `Practical Examination`
       - Question: Are these complementary (Education = grading, university_erp = exam logistics) or duplicative?

       **Attendance overlap:**
       - Education: `Student Attendance` doctype
       - university_erp: `Hostel Attendance`, `Biometric Attendance Log` -- plus faculty_api.submit_attendance creates `Student Attendance` records
       - Question: Does university_erp create records in Education's Student Attendance table?

       **Student data overlap:**
       - Education: `Student` doctype with programs, courses, enrollment
       - university_erp: Overrides at `overrides/student.py`, custom fields on Student
       - Question: What does the override add/change?

    2. For each overlap area, determine:
       - **Authoritative source**: Which app owns the primary logic?
       - **Integration pattern**: Does one extend the other, or do they conflict?
       - **Risk**: Could they produce inconsistent data?
       - **Recommendation**: Keep both, deprecate one, merge logic

    3. Read key source files to make informed determinations:
       - `university_erp/overrides/fees.py`
       - `university_erp/overrides/student.py`
       - `university_erp/overrides/assessment_result.py`
       - `university_erp/hooks.py` (for doc_events)

    4. Update DUPLICATE_AUDIT.md (append to existing file, which has doctype-level duplicates):
       - Add `## Business Logic Overlaps` section
       - For each overlap area: table with Education entities, university_erp entities, authoritative source, integration pattern, risk, recommendation
       - Add `## Authoritative Source Summary` table: | Domain | Authoritative App | Secondary App | Pattern |
