    **Phase A: Scan** -- Create `audit_sql_injection.py` and execute it.

    1. Walk `university_erp/university_erp/` directory tree.
    2. For each `.py` file, read content and search for dangerous patterns:
       - **Pattern A (f-string)**: regex `r'frappe\.db\.sql\s*\(\s*f["\']'` -- f-string directly in frappe.db.sql()
       - **Pattern B (f-string variable)**: Look for variables assigned f-strings that are later passed to `frappe.db.sql()`. Use a two-pass approach: (1) find all `frappe.db.sql(variable_name` calls, (2) check if `variable_name` was assigned via f-string.
       - **Pattern C (% format)**: regex `r'frappe\.db\.sql\s*\(\s*["\'].*%s.*["\']\s*%\s'` -- old-style % formatting
       - **Pattern D (.format())**: regex `r'frappe\.db\.sql\s*\(.*\.format\('` -- .format() calls
    3. For each finding, record: file_path, line_number, the offending line, pattern type.
    4. Classify severity:
       - CRITICAL: f-string/format with user-provided filter values (from function parameters named `filters`)
       - WARNING: f-string/format with internal values (from frappe.db lookups)
       - INFO: f-string/format used only for table names or column names (less exploitable but still bad practice)
    5. Generate SQL_AUDIT.md:
       - Summary: files scanned, total findings, by severity
       - Table: | # | File | Line | Severity | Pattern | Code Snippet |
       - Recommendations section

    6. `def run():` entry point.

    **Execute scan:**
    ```bash
    cd /workspace/development/frappe-bench
    bench --site university.local execute university_erp.scripts.audit_sql_injection.run
    ```

    **Phase B: Fix all CRITICAL and WARNING findings.**

    Per CONTEXT.md locked decision: "Fix any SQL injection risks found during scan."

    After the scan produces SQL_AUDIT.md, read the findings and fix each CRITICAL and WARNING item:
    - For each finding, open the source file and convert the f-string/format/% interpolation to parameterized queries.
    - **f-string pattern**: `frappe.db.sql(f"SELECT ... WHERE x = '{value}'")` becomes `frappe.db.sql("SELECT ... WHERE x = %(value)s", {"value": value})`
    - **% pattern**: `frappe.db.sql("SELECT ... WHERE x = '%s'" % value)` becomes `frappe.db.sql("SELECT ... WHERE x = %(value)s", {"value": value})`
    - **.format() pattern**: `frappe.db.sql("SELECT ... WHERE x = '{}'".format(value))` becomes `frappe.db.sql("SELECT ... WHERE x = %(value)s", {"value": value})`
    - For table/column name interpolation (INFO severity): convert to use `frappe.qb` query builder or `frappe.scrub()` for safe identifier handling. If the table/column name comes from a controlled source (hardcoded list, frappe.get_meta), INFO items may be left as-is but documented.
    - CRITICAL and WARNING items MUST be fixed. INFO items should be fixed if straightforward; document any left unfixed with justification.

    **After fixes, re-run the scan** to confirm zero CRITICAL and zero WARNING findings:
    ```bash
    cd /workspace/development/frappe-bench
    bench --site university.local execute university_erp.scripts.audit_sql_injection.run
    ```

    Update SQL_AUDIT.md with a "Remediation Status" section showing before/after counts and which files were fixed.
  </action>
