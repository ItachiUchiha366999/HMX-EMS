# Phase 03.1: Cross-App Integration Audit & Accessibility Testing - Research

**Researched:** 2026-03-18
**Domain:** Cross-app doctype analysis, link integrity, report verification, WCAG 2.1 AA accessibility
**Confidence:** HIGH

## Summary

This phase is an inserted audit phase covering four distinct work streams: (1) duplicate doctype detection across 5 apps (Frappe, ERPNext, Education, HRMS, university_erp), (2) cross-app Link field integrity testing, (3) verification of 59 reports and 27 faculty API endpoints, and (4) WCAG 2.1 AA accessibility audit with fixes for critical/major issues.

The codebase investigation reveals **27 exact-name doctype duplicates** between ERPNext and university_erp (primarily inventory/asset/procurement doctypes that university_erp re-implemented), plus 1 overlap with Education (quiz_question) and 1 with HRMS (grievance_type). Semantic overlaps are also significant -- Education has its own fee/assessment system while university_erp has parallel structures. The university_erp app has **268 doctypes** across **21 modules**, with **59 report Python files** and **58 report JSON definitions**. The faculty API has exactly **27 @frappe.whitelist() endpoints**. The portal-vue app has **33 Vue components** across shared (8) and faculty (22+) directories, all testable with the existing Vitest infrastructure (22 test files, jsdom environment).

**Primary recommendation:** Structure this as 4 independent work streams (duplicate audit, connection audit, report/API audit, accessibility audit), each producing a script + markdown report. Use `frappe.get_meta()` for schema comparison, `frappe.db.sql()` for link integrity, `frappe.desk.query_report.run()` for report testing, and axe-core (via @axe-core/cli or Lighthouse CLI) for accessibility scanning.

### Pattern 1: Doctype Schema Comparison via frappe.get_meta()
**What:** Use frappe.get_meta(doctype_name) to get full schema including fields, field types, options (link targets), istable flag, and module.
**When to use:** For all schema comparisons in the duplicate audit.
**Example:**
```python
# frappe.get_meta() returns a Meta object with these key attributes:
meta = frappe.get_meta("Student")
# meta.fields -> list of field dicts: {fieldname, fieldtype, options, label, ...}
# meta.istable -> 1 if child table
# meta.issingle -> 1 if single doctype
# meta.module -> "Education" or "University ERP" etc.

# To find all Link fields pointing to other apps:
for field in meta.fields:
    if field.fieldtype == "Link" and field.options:
        target_meta = frappe.get_meta(field.options)
        if target_meta.module != meta.module:
            # Cross-app link found
            pass
```

### Pattern 2: Discovering All Doctypes Per App
**What:** Use frappe.get_all("DocType") with module filters, or scan JSON files on disk.
**When to use:** For building the complete doctype inventory.
**Example:**
```python
# Method 1: Database query (requires site context)
all_doctypes = frappe.get_all("DocType",
    filters={"module": ["in", ["Education", "University ERP"]]},
    fields=["name", "module", "istable", "issingle"]
)

# Method 2: File system scan (works without DB, good for schema-only comparison)
import os, json
def get_doctypes_from_app(app_path):
    doctypes = []
    for root, dirs, files in os.walk(app_path):
        for f in files:
            if f.endswith('.json') and '/doctype/' in root:
                fpath = os.path.join(root, f)
                with open(fpath) as fp:
                    data = json.load(fp)
                    if data.get("doctype") == "DocType":
                        doctypes.append(data)
    return doctypes
```

### Anti-Patterns to Avoid
- **Manual doctype lists:** Never manually curate which doctypes to audit. Discover them programmatically from the file system or database.
- **Skipping reports that need filters:** Some reports require academic_year or department. Try empty filters first, then try with the most recent Academic Year / first Department as fallback.
- **Testing accessibility on dev-mode pages only:** Frappe desk in development mode loads extra debug toolbars. Test against production-built assets.

## Common Pitfalls

### Pitfall 1: Module vs App Confusion
**What goes wrong:** Frappe uses "module" as an organizational concept within apps. The same app can have multiple modules. When filtering doctypes, filtering by app name won't work -- you need module names.
**Why it happens:** university_erp has 21 modules. Education has 1 module ("Education"). ERPNext has dozens of modules.
**How to avoid:** Map modules to apps first. For university_erp: all 21 module names from modules.txt. For Education: module="Education". For HRMS: multiple modules. Build a module->app mapping dictionary.
**Warning signs:** Missing doctypes in audit results (filtered too narrowly).

### Pitfall 2: Exact Name Match False Positives
**What goes wrong:** university_erp has doctypes like `batch`, `asset`, `warehouse` that share names with ERPNext doctypes. In Frappe, only ONE doctype with a given name can exist in the database -- the second app's JSON is dead code or an override.
**Why it happens:** university_erp may have copied ERPNext doctype JSONs to customize them, or they genuinely conflict.
**How to avoid:** Check `frappe.db.get_value("DocType", doctype_name, "module")` to see which module owns the live version. Then compare the JSON definitions on disk.
**Warning signs:** 27 exact-name overlaps between ERPNext and university_erp is very high -- many are likely inventory/asset doctypes that university_erp re-created.

### Pitfall 3: Reports Requiring Mandatory Filters
**What goes wrong:** Running `frappe.desk.query_report.run("Report Name", filters={})` throws a validation error because the report requires academic_year, department, or date range.
**Why it happens:** Many reports have mandatory filter fields.
**How to avoid:** First try empty filters. On failure, read the report's `.js` file to find mandatory filters, then supply sensible defaults (current academic year, first department, last 30 days).
**Warning signs:** High error rate in initial report audit run.

### Pitfall 4: Accessibility Scan on Unauthenticated Pages
**What goes wrong:** axe-core/Lighthouse scan hits the login page instead of the actual portal content.
**Why it happens:** Portal pages require authentication. CLI tools don't have session cookies.
**How to avoid:** For portal-vue: scan the built SPA shell (it's a single page app -- scan the HTML + rendered components). For Frappe desk: use Lighthouse with `--extra-headers` to pass session cookies, or use puppeteer to log in first.
**Warning signs:** All pages show identical minimal HTML (login form).

### Pitfall 5: Dynamic Link Fields
**What goes wrong:** Missing Dynamic Link fields in cross-app audit. Dynamic Links have `fieldtype: "Dynamic Link"` and reference a companion field that holds the doctype name.
**Why it happens:** Dynamic Links are a two-field pattern: one field holds the doctype name, another holds the document name. Easy to miss in automated scanning.
**How to avoid:** When scanning for cross-app links, also scan for `fieldtype == "Dynamic Link"` and resolve the companion `options` field to find the doctype source field.

## Code Examples

### Complete Doctype Discovery Pattern
```python
import frappe
import json
import os

def get_app_doctype_map():
    """Build a mapping of app -> module -> [doctypes] from the file system."""
    apps_path = frappe.get_app_path("frappe").replace("/frappe/frappe", "")
    app_names = ["frappe", "erpnext", "education", "hrms", "university_erp"]

    app_map = {}
    for app in app_names:
        app_map[app] = {}
        app_path = os.path.join(apps_path, app)
        for root, dirs, files in os.walk(app_path):
            for f in files:
                if not f.endswith(".json") or "/doctype/" not in root:
                    continue
                fpath = os.path.join(root, f)
                try:
                    with open(fpath) as fp:
                        data = json.load(fp)
                    if data.get("doctype") != "DocType":
                        continue
                    module = data.get("module", "Unknown")
                    name = data.get("name", f.replace(".json", ""))
                    if module not in app_map[app]:
                        app_map[app][module] = []
                    app_map[app][module].append({
                        "name": name,
                        "fields": data.get("fields", []),
                        "istable": data.get("istable", 0),
                        "issingle": data.get("issingle", 0),
                    })
                except (json.JSONDecodeError, KeyError):
                    pass
    return app_map
```

### Cross-App Link Discovery
```python
def find_cross_app_links(app_map):
    """Find all Link fields that reference doctypes in other apps."""
    # Build doctype->app lookup
    dt_to_app = {}
    for app, modules in app_map.items():
        for module, doctypes in modules.items():
            for dt in doctypes:
                dt_to_app[dt["name"]] = app

    cross_links = []
    for app, modules in app_map.items():
        for module, doctypes in modules.items():
            for dt in doctypes:
                for field in dt["fields"]:
                    if field.get("fieldtype") == "Link" and field.get("options"):
                        target = field["options"]
                        target_app = dt_to_app.get(target)
                        if target_app and target_app != app:
                            cross_links.append({
                                "source_app": app,
                                "source_doctype": dt["name"],
                                "field": field["fieldname"],
                                "target_app": target_app,
                                "target_doctype": target,
                            })
    return cross_links
```

### Report Batch Runner
```python
def audit_all_reports():
    """Run all Script Reports and check for errors."""
    from frappe.desk.query_report import run as run_report

    reports = frappe.get_all("Report",
        filters={"report_type": ["in", ["Script Report", "Query Report"]], "disabled": 0},
        fields=["name", "module", "report_type"]
    )

    results = []
    for report in reports:
        try:
            result = run_report(report["name"], filters={})
            has_data = bool(result.get("result"))
            results.append({
                "report": report["name"],
                "module": report["module"],
                "status": "PASS",
                "rows": len(result.get("result", [])),
                "columns": len(result.get("columns", [])),
            })
        except Exception as e:
            results.append({
                "report": report["name"],
                "module": report["module"],
                "status": "ERROR",
                "error": str(e)[:200],
            })
    return results
```

## Open Questions

1. **Which version of each duplicate doctype is "live" in the database?**
   - What we know: Frappe can only have one DocType with a given name. When two apps define the same name, the last-installed wins.
   - What's unclear: Which app's version is active for the 27 ERPNext<>university_erp overlaps.
   - Recommendation: The audit script should check `frappe.db.get_value("DocType", name, "module")` for each duplicate to determine the live owner.

2. **How to authenticate axe-core CLI for portal pages?**
   - What we know: portal-vue requires login. axe-core CLI supports `--disable` flags but not cookie injection natively.
   - What's unclear: Whether the portal SPA can be scanned in an unauthenticated state for HTML/CSS issues.
   - Recommendation: Use @axe-core/cli on the built HTML shell (index.html) for static checks. For full page scanning, use Lighthouse with puppeteer login script or test against the dev server with a hardcoded session.

3. **How many reports actually need specific filters to execute?**
   - What we know: Some reports have mandatory filters (academic year, department).
   - What's unclear: Exactly how many of the 59 reports will fail with empty filters.
   - Recommendation: First pass with empty filters to categorize; second pass with sensible defaults for those that fail.
