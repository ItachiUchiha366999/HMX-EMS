<decisions>
## Implementation Decisions

### Custom Field Migration (ROOT CAUSE FIX)
- **Employee custom fields**: 13 fields from `employee_custom_fields.json` — includes custom_is_faculty, custom_employee_category, custom_faculty_profile, custom_faculty_type, custom_current_workload, qualifications table, experience fields
- **Leave Application custom fields**: 9 fields from `leave_application_custom_fields.json` — includes custom_total_classes_affected, custom_affects_teaching, custom_substitute_arranged, custom_affected_courses table
- **Method**: Load fixture JSONs programmatically via bench console script, then run bench migrate
- **Verification**: After migration, re-run faculty API audit and report audit

### Report Fixes (10 university_erp reports)
- Fix SQL column errors: s.program, ff.title, pjo.package_offered, pa.program, g.submission_date, fr.reason, e.custom_is_faculty, e.custom_employee_category, la.custom_total_classes_affected
- Fix broken module imports: examination_result_analysis, program_wise_fee_report
- Fix missing settings field: library_fine_per_day on University Settings
- Fix executive dashboard: get_overview_metrics() Unknown column 'status' on Student table

### Comprehensive API Testing (NEW)
- **Scope**: Discover ALL @frappe.whitelist() endpoints across university_erp app. Not just faculty_api — include portal_api, analytics API, health_check, all module APIs.
- **Method**: Automated endpoint discovery via AST/regex scan of .py files, then programmatic invocation with appropriate user context.
- **Test per endpoint**: Does it return valid response? Does it respect permissions? Does it handle missing data gracefully?

### Permission Testing (NEW)
- **Roles to test**: System Manager (admin), Faculty (instructor), Student, Guardian (parent), HOD, University VC, University Dean, University Registrar, University Finance Officer
- **What to test per role**: Can access allowed doctypes, cannot access restricted doctypes, API endpoints respect role checks, portal routes match role permissions
- **Method**: Script that switches frappe.session.user to test users with each role, then attempts CRUD + API access
- **Output**: PERMISSION_AUDIT.md with pass/fail matrix per role × doctype/endpoint

### Desk UI Verification (NEW)
- **Scope**: All Frappe workspace pages, module sidebar links, desk dashboard pages, report links in desk
- **What to test**: Every link resolves (no 404s), every workspace card links to valid doctype/page, every dashboard renders without JS errors
- **Method**: Automated via Frappe desk API (list workspaces, list shortcuts, verify each target exists)
- **Output**: DESK_AUDIT.md

### Portal UI Verification (NEW)
- **Scope**: All portal-vue routes (faculty pages built in Phase 3)
- **What to test**: Each route renders without console errors, API calls succeed, no broken component imports
- **Method**: Vitest component render tests (extend existing a11y test suite)
- **Output**: PORTAL_AUDIT.md

### SQL Query Scan (NEW)
- **Scope**: All .py files in university_erp that contain frappe.db.sql or raw SQL
- **What to test**: No f-string interpolation in queries (injection risk), all referenced columns exist in DB schema
- **Method**: AST scan for f-string + frappe.db.sql patterns, plus schema validation against live DB
- **Output**: SQL_AUDIT.md

### Duplicate Business Logic Audit (EXPANDED)
- **Beyond doctype names**: Audit overlapping BUSINESS LOGIC — e.g., Fees in Education vs Fee Payment in university_erp, Exams in Education vs Exams in university_erp
- **Determine**: Which app's implementation is authoritative? Are there conflicting behaviors?
- **Output**: Update DUPLICATE_AUDIT.md with business logic overlap section

### Fix Everything
- **Fix all custom field migrations** (unblocks faculty API + 4 reports)
- **Fix all 10 broken reports** (SQL + import fixes)
- **Fix executive dashboard** status column
- **Fix any permission gaps** found during audit
- **Fix any SQL injection risks** found during scan
- **Fix any broken desk/portal links** found during verification
- **Re-run ALL audits** after fixes to verify green

### Claude's Discretion
- Exact order of fixes (suggest: custom fields first since they unblock the most)
- Test user creation strategy (new test users vs existing demo users)
- How to handle ERPNext/HRMS reports that fail due to missing company setup (document as known, not fix)
- Whether to add permanent regression test scripts or keep as one-time audits

</decisions>
