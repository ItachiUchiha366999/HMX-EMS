# Phase 03.1: Cross-App Integration Audit & Accessibility Testing - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Comprehensive audit and testing of the entire university ERP ecosystem. Four deliverables: (1) Duplicate doctype audit across all 4 apps with full schema diff, (2) Cross-app connection integrity testing of all link fields, (3) Report and dashboard data flow verification including all 54+ reports and 27 faculty API endpoints, (4) UI accessibility audit against WCAG 2.1 AA with fixes for critical/major issues. This phase produces audit reports AND fixes critical issues found — it is not documentation-only.

</domain>

<decisions>
## Implementation Decisions

### Duplicate Doctype Audit
- **App scope**: All four apps — Education, ERPNext, HRMS, university_erp. Full cross-product comparison.
- **Depth**: Full schema diff — compare fields, field types, link targets, child tables. Not just name matching.
- **Scope extras**: Include child tables (istable=1), virtual doctypes, and single doctypes. Full picture.
- **Script type**: One-time bench console script (`.py` run via `bench --site university.local execute`). Not a permanent management command.
- **Output**: `DUPLICATE_AUDIT.md` report grouped by overlap type — Exact Name Matches, Semantic Overlaps (different names, similar fields), Shared Link Targets. Tables with app, doctype, field count, recommendation.
- **Action on duplicates**: Document + recommend (keep, deprecate, merge, alias). No code changes to resolve duplicates in this phase — just the audit and recommendations.

### Cross-App Connection Testing
- **Scope**: All Link and Dynamic Link fields that reference doctypes in other apps. Automatic discovery — not a manually curated list.
- **What to test**: Verify every cross-app link resolves to a real record. Count broken links per doctype pair.
- **Severity levels**: Broken link (field points to non-existent record) = FAILURE. Orphan record (no incoming links) = WARNING.
- **Output**: Script prints live results to console AND writes `CONNECTION_AUDIT.md` with tables: link field, source app, target app, total records, broken links count, broken link details (first 10 per type).

### Report & Dashboard Data Flow
- **Report scope**: ALL 54+ reports. Run each via `frappe.desk.query_report.run` with default filters. Verify each returns data or graceful empty (no errors).
- **Faculty API testing**: Integration-test all 27 `faculty_api.py` endpoints with a real faculty user session. Verify response structure, data types, and cross-app data resolution.
- **Dashboard testing**: Call each dashboard endpoint, compare returned values against direct SQL queries on source tables. Verify aggregation accuracy, not just non-empty responses.
- **Output**: `REPORT_AUDIT.md` and `API_AUDIT.md` with pass/fail per endpoint and details for failures.

### UI Accessibility
- **Standard**: WCAG 2.1 AA compliance.
- **Scope**: Everything — portal-vue app, Frappe desk customizations (university_desk.css, portal.css), and Frappe desk upstream UI. Full coverage.
- **Testing method**: Automated only — axe-core/Lighthouse scans on all portal pages and key Frappe desk pages.
- **Output + action**: `ACCESSIBILITY_AUDIT.md` report listing all issues by severity. Then fix all critical and major issues in this same phase. Minor issues documented for later.

### Claude's Discretion
- Exact script structure and helper functions
- How to authenticate for API integration tests (create test user or use existing faculty)
- Which Frappe desk pages to include in accessibility scan
- How to handle reports that require specific filter values to return data
- Severity classification for edge cases

</decisions>

## Existing Code Insights

### Reusable Assets
- **health_check.py**: Already has cross-app check patterns (Student↔Enrollment, Fees↔GL, Employee↔Faculty, Academic Year). Can extend or complement.
- **portal_api.py + faculty_api.py**: Established `@frappe.whitelist()` patterns for API endpoints.
- **Vitest infrastructure**: 102 tests across 22 files — test runner and config already set up.

### Established Patterns
- **frappe.db.sql()**: Parameterized queries (fixed in Phase 1). All new scripts should follow.
- **frappe.get_meta()**: Standard way to read doctype schemas programmatically.
- **frappe.desk.query_report.run()**: Standard Frappe report runner API.

### Integration Points
- Audit scripts run via `bench --site university.local execute <script_path>`
- Reports run via Frappe's query_report API
- API tests can use `requests` library with session cookies or `frappe.call()` from bench console
- Accessibility scans need a running server at localhost:8000 (internal) / localhost:18000 (host)
