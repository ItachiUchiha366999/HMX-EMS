<decisions>
## Implementation Decisions

### Duplicate Doctype Audit
- **App scope**: All four apps â€” Education, ERPNext, HRMS, university_erp. Full cross-product comparison.
- **Depth**: Full schema diff â€” compare fields, field types, link targets, child tables. Not just name matching.
- **Scope extras**: Include child tables (istable=1), virtual doctypes, and single doctypes. Full picture.
- **Script type**: One-time bench console script (`.py` run via `bench --site university.local execute`). Not a permanent management command.
- **Output**: `DUPLICATE_AUDIT.md` report grouped by overlap type â€” Exact Name Matches, Semantic Overlaps (different names, similar fields), Shared Link Targets. Tables with app, doctype, field count, recommendation.
- **Action on duplicates**: Document + recommend (keep, deprecate, merge, alias). No code changes to resolve duplicates in this phase â€” just the audit and recommendations.

### Cross-App Connection Testing
- **Scope**: All Link and Dynamic Link fields that reference doctypes in other apps. Automatic discovery â€” not a manually curated list.
- **What to test**: Verify every cross-app link resolves to a real record. Count broken links per doctype pair.
- **Severity levels**: Broken link (field points to non-existent record) = FAILURE. Orphan record (no incoming links) = WARNING.
- **Output**: Script prints live results to console AND writes `CONNECTION_AUDIT.md` with tables: link field, source app, target app, total records, broken links count, broken link details (first 10 per type).

### Report & Dashboard Data Flow
- **Report scope**: ALL 279 registered reports across all apps. Run each via `frappe.desk.query_report.run` with default filters. Verify each returns data or graceful empty (no errors).
- **Faculty API testing**: Integration-test all 27 `faculty_api.py` endpoints with a real faculty user session. Verify response structure, data types, and cross-app data resolution.
- **Dashboard testing**: Call each dashboard endpoint, compare returned values against direct SQL queries on source tables. Verify aggregation accuracy, not just non-empty responses.
- **Output**: `REPORT_AUDIT.md` and `API_AUDIT.md` with pass/fail per endpoint and details for failures.

### UI Accessibility
- **Standard**: WCAG 2.1 AA compliance.
- **Scope**: Everything â€” portal-vue app, Frappe desk customizations (university_desk.css, portal.css), and Frappe desk upstream UI. Full coverage.
- **Testing method**: Automated only â€” axe-core/Lighthouse scans on all portal pages and key Frappe desk pages.
- **Output + action**: `ACCESSIBILITY_AUDIT.md` report listing all issues by severity. Then fix all critical and major issues in this same phase. Minor issues documented for later.
