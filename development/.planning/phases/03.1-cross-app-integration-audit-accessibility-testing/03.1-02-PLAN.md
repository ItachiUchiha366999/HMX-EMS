Create `frappe-bench/apps/university_erp/scripts/audit_faculty_api.py` callable via `bench --site university.local execute university_erp.scripts.audit_faculty_api.run`.

The script must:

1. **Find a valid faculty user** for testing:
   - Query: `frappe.db.sql("SELECT e.user_id FROM tabEmployee e WHERE e.user_id IS NOT NULL AND e.user_id != '' AND e.custom_is_faculty = 1 LIMIT 1")`
   - If no faculty found, try: `frappe.db.sql("SELECT e.user_id FROM tabEmployee e WHERE e.user_id IS NOT NULL AND e.user_id != '' LIMIT 1")` as fallback
   - If still none found, log error and exit gracefully
   - Set session: `frappe.set_user(faculty_user)`

2. **Discover all endpoints programmatically**: Read faculty_api.py source, find all functions decorated with `@frappe.whitelist()`. Do NOT hardcode the 27 endpoint names — discover them by importing the module and inspecting `dir()` or by regex on the source file. This future-proofs against added/removed endpoints.

3. **Test each GET endpoint** (functions starting with `get_`):
   - Call with no arguments first
   - If fails with missing argument error, try to supply reasonable defaults:
     - `course`: first Course from `frappe.get_all("Course", limit=1)`
     - `student`: first Student from `frappe.get_all("Student", limit=1)`
     - `student_group`: first Student Group from `frappe.get_all("Student Group", limit=1)`
     - `date`: today's date
     - `academic_year`: most recent Academic Year
   - Record: endpoint name, status (PASS/FAIL/DEGRADED), response type (dict/list/None), response size, error message
   - DEGRADED = endpoint executes but returns empty data (no exception, but empty result)

4. **Test each ACTION endpoint** (submit_attendance, save_grades, approve/reject leave, etc.):
   - Do NOT actually submit/save data — just verify the function can be called and validates inputs
   - Call with obviously invalid/empty data to verify it returns a validation error (not a crash/traceback)
   - Record: endpoint name, expected validation error, actual response

5. **Verify cross-app data resolution** for key endpoints:
   - `get_faculty_dashboard`: Check that returned pending_tasks references real doctypes (Student Attendance, Assessment Result, etc.)
   - `get_today_classes`: Check that returned class data includes room, course, and time from Teaching Assignment Schedule
   - `get_students_for_grading(course)`: Check that returned students have valid student IDs that exist in Student doctype
   - `get_leave_balance`: Check that returned leave types exist in Leave Type doctype

6. **Generate API_AUDIT.md** with sections:
   - Executive Summary (total endpoints, pass, fail, degraded counts)
   - Endpoint Discovery (how many found, list of all discovered endpoint names)
   - GET Endpoint Results table: | # | Endpoint | Status | Response Type | Data Size | Cross-App Check | Notes |
   - ACTION Endpoint Results table: | # | Endpoint | Validation Check | Status | Notes |
   - Cross-App Data Resolution table: | Endpoint | Referenced Doctype | Exists in DB? | Records Found? |
   - Recommendations (endpoints that need fixing, data gaps, etc.)

7. **Reset user context** at the end: `frappe.set_user("Administrator")`
8. Write to `.planning/phases/03.1-cross-app-integration-audit-accessibility-testing/API_AUDIT.md`.
